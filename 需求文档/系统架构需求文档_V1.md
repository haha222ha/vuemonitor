## 一、项目定位

### 1.1 产品本质

**不是"爬虫工具"，而是"AI数据生态系统"**

- 本地客户端负责 **"数据入口"** — 采集与特征提取
- 云端服务负责 **"AI决策大脑"** — 分析与预测
- 特征引擎负责 **"数据炼金"** — 原始数据转化为结构化特征

### 1.2 商业模式

```
免费监控工具（拉新） → 数据沉淀 → AI增强分析 → 付费转化
```

| 层级 | 定位 | 目的 |
|------|------|------|
| 免费层 | 本地监控助手（Hook Product） | 拉新、形成使用习惯 |
| 付费层 | AI决策分析平台 | 商业转化、持续收入 |

### 1.3 架构模式

**SaaS + Edge Client（边缘客户端）**

- 云端不做集中式大规模采集
- 本地客户端完成数据采集与特征提取
- 云端只接收匿名特征值，做智能分析

## 三、代理IP与反检测模块

### 3.1 为什么需要这个模块？

Electron 客户端在本地进行页面采集时，高频访问同一平台可能触发反爬机制。需要一套代理IP与反检测策略来保障采集稳定性。

### 3.2 系统定位

> ❗这不是"大规模爬虫的代理池"，而是"用户本地监控助手的反检测保障"

核心区别：
- 不做集中式大规模采集，因此不需要海量代理IP
- 每个用户只监控自己关注的少量商品，访问频率低
- 代理IP是"保障性"功能，不是"攻击性"功能

### 3.3 代理IP策略设计

#### 代理类型选择

| 代理类型 | 适用场景 | 成本 | 推荐度 |
|----------|----------|------|--------|
| 用户自有代理 | 高级用户自行配置 | 用户承担 | ★★★★★ |
| 住宅代理（Residential） | 模拟真实用户访问 | 较高 | ★★★★ |
| 数据中心代理 | 低频采集兜底 | 较低 | ★★★ |
| 无代理直连 | 低频监控（默认） | 免费 | ★★★★★ |

#### 代理配置架构

```
用户代理配置
   ↓
代理管理器（Proxy Manager）
   ↓
┌─────────────────────────┐
│  代理池（用户自配 / 内置）│
│  ├─ 代理轮换策略          │
│  ├─ 健康检测              │
│  ├─ 失败自动切换          │
│  └─ 代理可用性统计        │
└─────────────────────────┘
   ↓
Electron 采集 Worker
```

#### 代理轮换策略

| 策略 | 说明 |
|------|------|
| 顺序轮换 | 按列表顺序依次使用 |
| 随机轮换 | 随机选择可用代理 |
| 故障切换 | 当前代理失败自动切换下一个 |
| 智能调度 | 根据代理响应速度和成功率加权选择 |

### 3.4 反检测策略

#### 浏览器指纹伪装

| 维度 | 策略 | 说明 |
|------|------|------|
| User-Agent | 随机轮换 | 使用常见浏览器UA |
| 屏幕分辨率 | 随机生成 | 模拟不同设备 |
| WebGL指纹 | 轻微扰动 | 避免指纹唯一 |
| Canvas指纹 | 轻微噪声 | 防止Canvas指纹追踪 |
| 字体列表 | 随机裁剪 | 模拟不同系统字体 |
| 时区/语言 | 匹配代理IP地区 | 保持一致性 |

#### 行为模拟

| 策略 | 说明 |
|------|------|
| 随机延迟 | 页面操作间加入随机等待时间 |
| 模拟滚动 | 模拟真实用户的滚动行为 |
| 鼠标轨迹 | 可选的鼠标移动模拟 |
| 访问间隔 | 同一商品两次采集之间保持合理间隔 |

#### 请求频率控制

| 场景 | 建议频率 | 说明 |
|------|----------|------|
| 单商品监控 | 30分钟 ~ 2小时 | 根据用户配置 |
| 同域名总频率 | ≤ 10次/小时 | 避免触发频率限制 |
| 高峰期 | 自动降频 | 检测到异常时主动降低频率 |

### 3.5 代理IP功能分级

| 会员等级 | 代理能力 |
|----------|----------|
| Free | 直连（无代理），基础频率限制 |
| Pro | 支持配置自有代理，智能频率控制，反检测增强 |
| Enterprise | 内置代理池，团队共享代理配置 |

## 四、特征引擎层（Feature Engine）

### 4.1 为什么需要这一层？

**原始数据 ≠ Intelligence**

AI 是否准确，不取决于原始数据量，而取决于 **特征质量**。

```
❌ 之前：采集 → AI（直接把原始数据丢给AI）

✅ 升级：采集 → 特征引擎 → AI决策（先提炼特征，再交给AI）
```

### 4.2 原始数据 vs 特征数据

**❌ 原始数据（AI不直接需要）**：
```json
{
  "sales": [12, 20, 55, 90]
}
```

**✅ AI真正需要的特征数据**：
```json
{
  "growth_rate": 2.3,
  "acceleration": 1.8,
  "volatility": 0.2,
  "competition_index": 0.6
}
```

### 4.3 Feature Engine 职责

#### 数据清洗

| 功能 | 说明 |
|------|------|
| 去噪 | 过滤异常波动数据 |
| 去异常 | 剔除明显错误采集值 |
| 标准化 | 统一数据格式与量纲 |
| 缺失值处理 | 插值或标记缺失数据 |

#### 特征提取

| 特征 | 说明 | 示例 |
|------|------|------|
| 增长率（growth_rate） | 销量增长速度 | 2.3 表示增长130% |
| 加速度（acceleration） | 增长趋势的变化率 | 1.8 表示增长在加速 |
| 波动率（volatility） | 数据稳定性 | 0.2 表示较稳定 |
| 竞争强度（competition_index） | 同类竞品压力 | 0.6 表示中等竞争 |
| 生命周期阶段 | 商品所处阶段 | 导入期/成长期/成熟期/衰退期 |

#### 趋势计算

| 趋势类型 | 说明 |
|----------|------|
| 短期趋势 | 近7天变化方向与力度 |
| 长期趋势 | 近30天/90天变化方向 |
| 热度加速度 | 关注度增长的变化率 |

### 4.4 Feature Engine 部署位置

| 部分 | 部署位置 | 职责 |
|------|----------|------|
| 本地特征引擎 | Electron 客户端 | 单用户数据清洗、基础特征提取、本地趋势计算 |
| 云端特征引擎 | FastAPI 服务端 | 群体行为聚合、跨用户趋势计算、全局特征增强 |

## 五、数据流转策略

### 5.1 数据分类与流向

| 数据类型 | 存储位置 | 是否上传 | 说明 |
|----------|----------|----------|------|
| 完整采集记录（原始数据） | 用户本地 | ❌ 不上传 | 每日销量、原始页面、DOM数据 |
| 单用户特征值 | 用户本地 + 云端 | ✅ 上传匿名特征 | 增长率、波动率、竞争强度等 |
| 匿名聚合趋势数据 | 云端 | ✅ 自动聚合 | 类目热度、关注人数、趋势方向 |
| AI分析结果 | 云端 → 推送到客户端 | ✅ | 爆品预测、风险分析、决策建议 |

### 5.2 上传数据示例

**单用户匿名特征值**：
```json
{
  "product_id": "123",
  "category": "美妆",
  "growth_rate": 1.8,
  "views_growth": 2.1,
  "competition_index": 0.6,
  "volatility": 0.15,
  "lifecycle_stage": "growth"
}
```

**群体聚合趋势数据**：
```json
{
  "category": "美妆",
  "growth_rate": 1.8,
  "watch_count": 126,
  "trend": "up",
  "heat_acceleration": 0.35
}
```

### 5.3 核心原则

> **不上传海量原始数据，只上传匿名特征值**

系统性质从 **"集中式采集平台"** 变为 **"AI分析平台"**，风险等级大幅降低。

## 六、会员与授权体系

### 6.1 为什么用授权码替代支付系统？

当前阶段重点：**验证产品价值**，而非支付接入。

支付系统会瞬间增加以下复杂度：

| 复杂度维度 | 说明 |
|------------|------|
| 法务 | 支付合规、用户协议 |
| 风控 | 欺诈检测、异常交易 |
| 税务 | 发票、税务申报 |
| 退款 | 退款流程、争议处理 |
| 渠道 | 支付渠道对接与维护 |
| 回调 | 支付回调、状态同步 |
| 漏单 | 订单丢失处理 |
| 对账 | 资金对账、差异排查 |

> ❗用"授权码体系"替代"在线支付体系"是当前阶段非常合理的MVP路线。

## 七、权限控制层（Feature Gate）

### 7.1 为什么需要权限控制层？

> ❗系统里每个功能，都必须判断用户权限。支付只是收钱，权限体系才是真正的商业核心。

### 7.2 权限控制架构

```
API请求
   ↓
权限校验中间件（Feature Gate）
   ↓
┌─────────────────────┐
│  查询用户会员状态     │
│  ├─ plan（等级）      │
│  ├─ expires_at（到期）│
│  └─ 功能配额          │
└─────────────────────┘
   ↓
功能放行 / 拒绝
```

### 7.3 Feature Gate 功能门控清单

| 功能门 | Free | Pro | Enterprise |
|--------|------|-----|------------|
| `gate:monitor:max_items` | 5 | ∞ | ∞ |
| `gate:storage:cloud_backup` | ❌ | ✅ | ✅ |
| `gate:ai:daily_calls` | 3/天 | ∞ | ∞ |
| `gate:ai:report` | ❌ | ✅ | ✅ |
| `gate:ai:prediction` | ❌ | ✅ | ✅ |
| `gate:proxy:custom` | ❌ | ✅ | ✅ |
| `gate:proxy:built_in_pool` | ❌ | ❌ | ✅ |
| `gate:sync:multi_device` | ❌ | ✅ | ✅ |
| `gate:team:collaboration` | ❌ | ❌ | ✅ |
| `gate:api:access` | ❌ | ❌ | ✅ |

### 7.4 云端权限校验（FastAPI 中间件）

```python
# 伪代码示意
async def feature_gate(required_plan: str, required_feature: str):
    async def middleware(request):
        user = get_current_user(request)
        if not has_permission(user, required_plan, required_feature):
            raise PermissionDenied(feature=required_feature)
        return await call_next(request)
    return middleware
```

### 7.5 本地权限缓存（Electron 端）

**为什么需要本地缓存？** 用户不一定一直联网。

#### 本地缓存结构

```json
{
  "plan": "pro",
  "expires_at": "2026-12-01T00:00:00Z",
  "features": {
    "max_monitor_items": -1,
    "ai_daily_calls": -1,
    "cloud_backup": true,
    "ai_report": true,
    "custom_proxy": true
  },
  "last_verified_at": "2026-05-08T10:00:00Z"
}
```

#### 缓存策略

| 策略 | 说明 |
|------|------|
| 本地缓存有效期 | 24小时 |
| 联网时自动校验 | 每次启动 + 每24小时 |
| 离线宽容期 | 最多72小时离线使用 |
| 过期强制校验 | 超过72小时必须联网验证 |
| UI状态同步 | 根据权限显示/灰掉功能按钮 |

#### 本地权限UI表现

| 场景 | Free 用户 | Pro 用户 |
|------|-----------|----------|
| AI分析按钮 | 灰掉 + "升级Pro"提示 | 正常可用 |
| 监控数量超限 | 提示"已达上限，升级Pro" | 无限制 |
| 云备份开关 | 隐藏或灰掉 | 正常可用 |
| 代理IP配置 | 隐藏 | 正常可用 |

## 十、三大护城河

### 10.1 护城河一：群体行为趋势库

**大家最近在看什么**

- 匿名聚合的用户监控行为
- 类目热度变化趋势
- 跨用户选品偏好洞察
- 市场情绪指标

**壁垒来源**：用户规模越大，趋势数据越准确，越难被复制。

### 10.2 护城河二：特征工程体系

**如何理解趋势**

- 增长率、波动率、竞争强度的计算方法
- 生命周期判断算法
- 热度加速度模型
- 特征组合与权重体系

**壁垒来源**：AI模型容易换，但特征体系才是真正壁垒。特征质量决定AI上限。

### 10.3 护城河三：AI决策Prompt体系

**如何输出商业决策**

- AI分析Prompt模板库
- 标准化决策输出格式
- 行业特定的分析框架
- 用户反馈驱动的Prompt优化

**壁垒来源**：持续积累的商业决策知识，越用越准。

### 10.4 护城河对比

| 护城河 | 可复制性 | 积累效应 | 核心价值 |
|--------|----------|----------|----------|
| 群体行为趋势库 | 低（需用户规模） | 强（数据越多越准） | 市场洞察 |
| 特征工程体系 | 中（需持续迭代） | 强（越迭代越精准） | 理解趋势 |
| AI决策Prompt体系 | 中（需领域知识） | 强（越用越准） | 商业决策 |

## 十一、数据合规与用户信任策略

### 11.1 数据分类

| 类型 | 性质 | 处理方式 |
|------|------|----------|
| 用户私有数据（监控了哪些商品、选品思路） | 敏感 | 仅存本地，不上传 |
| 公共市场数据（类目热度、增长趋势） | 可聚合 | 匿名化后上传分析 |

### 11.2 用户告知机制

首次开启云同步时，需明确弹窗说明：

> **是否开启云端备份与匿名趋势分析？**
>
> 我们不会上传完整原始数据，仅会匿名化统计部分趋势特征，用于提升AI分析能力。

### 11.3 三原则

1. **明确告知** — 用户知道哪些数据会被上传
2. **用户可控** — 用户可以随时关闭云同步
3. **去敏感化** — 上传前移除所有个人标识信息

## 十七、核心原则（一句话总结）

> **Electron 负责"用户本地数据入口"，Feature Engine 负责"数据炼金"，FastAPI 负责"AI决策大脑"，Feature Gate 负责"商业边界"。**
>
> **本地客户端负责"采集"，特征引擎负责"提炼"，云端系统负责"智能"，权限体系负责"变现"。**
>
> **真正的护城河不是技术栈，而是：群体行为趋势库 + 特征工程体系 + AI决策Prompt体系。**
>
> **当前阶段用授权码替代支付，是合理的MVP路线。支付只是收钱，权限体系才是真正的商业核心。**
