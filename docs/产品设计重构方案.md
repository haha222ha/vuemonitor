### 1.2 职责划分

| 层级 | 职责 | 不做什么 |
|-----|------|---------|
| 服务端 | AI分析、特征计算、排名排序、告警评估、群体洞察、数据聚合 | 不做UI渲染 |
| 客户端 | 数据采集（唯一写入点）、数据展示、用户交互、离线降级 | 不重复做服务端计算 |
| 同步层 | push本地采集数据到云端、pull云端计算结果到本地 | 不做业务逻辑 |

### 1.3 数据流

```
本地采集 → 本地SQLite → 云端同步(push) → 服务端存储
                                              ↓
                                         服务端AI分析
                                         服务端特征计算
                                         服务端告警评估
                                              ↓
客户端展示 ← API调用 ← 服务端结果
           ← WebSocket实时推送
```

## 二、当前问题诊断

### 2.1 产品定位问题

| 问题 | 现状 | 目标 |
|-----|------|------|
| 首页是运维监控台 | 展示采集数/队列数/内存/并发 | 商业情报盘，展示机会/风险/趋势 |
| 导航是工具思维 | "数据盘面/商品监控/AI分析" | 决策思维 "机会雷达/我的商品/AI决策" |
| 服务端能力浪费 | 15+个API未被客户端使用 | 充分利用已有API |
| 告警系统重复 | 客户端本地规则+服务端告警引擎 | 统一使用服务端告警 |
| AI展示简陋 | 纯文本展示分析结果 | 结构化可视化展示 |

### 2.2 服务端能力使用现状

| 服务端能力 | API端点 | 客户端使用 |
|-----------|---------|----------|
| 商品排名+百分位 | `/feature/product-ranking/{id}` | ✅ ProductDetailView已使用 |
| 我的商品排名 | `/feature/product-rankings` | ❌ 未使用 |
| 品类统计 | `/feature/category-stats` | ❌ 未使用 |

| 行为模式 | `/feature/crowd/behavior-patterns` | ⚠️ 展示简陋 |
| 匿名价格基准 | `/feature/anonymous/price-benchmark` | ✅ ProductDetailView已使用 |
| 匿名销量基准 | `/feature/anonymous/sales-benchmark` | ✅ ProductDetailView已使用 |
| 趋势时间序列 | `/feature/crowd/trend-timeseries` | ❌ 未使用 |
| 告警规则引擎 | `/alert-rules/*` | ❌ 用的是本地规则 |
| 告警事件 | `/alert-rules/events/all` | ❌ 未使用 |
| AI 8种分析 | `/ai/analyze` | ⚠️ 部分使用 |

### 2.3 FOMO数据基础（真实数据驱动）

服务端已有能力可直接支撑真实FOMO：

1. **排名百分位** → "你的商品销量超过同类87%的商品"
2. **品类热力图** → "XX品类正在爆发，已有N个卖家在监控"
3. **匿名基准** → "同类商品均价¥89，你的定价偏高"
4. **行为模式** → "当前主流趋势：上升，最佳价格带：¥50-100"
5. **告警事件** → "3个商品价格异动，2个商品销量激增"

## 三、产品重构方案

### 3.1 导航重构

**当前导航 → 新导航**

```
当前:                          新:
核心功能                       机会雷达 (首页)
  数据盘面                       ├─ 今日机会榜
  商品监控                       ├─ 异动监控
  AI分析                         └─ 品类热力图
工具                           我的商品
  采集调度                       ├─ 商品列表（带排名百分位）
  监控规则                       ├─ 商品详情（带AI分析入口）
  通知中心                       └─ 商品对比（带匿名基准）
  商品对比                     AI决策
系统                             ├─ 快速分析（8种分析类型）
  设置                           ├─ 分析报告（4种报告类型）
  授权管理                       └─ 选品建议
                               通知中心
                               设置
```

### 3.2 首页重构：机会雷达

**首页第一屏必须回答用户三个问题：**

1. **今天有什么机会？** → 今日机会榜
2. **我的商品有什么异动？** → 异动监控
3. **市场在发生什么？** → 品类热力图

#### 顶部统计卡片

| 当前 | 新 | 数据来源 |
|-----|---|---------|
| 监控商品数 | 机会商品数 | `/feature/product-rankings` 排名前30% |
| 活跃采集数 | 今日趋势 | `/dashboard/stats` → `today_trend` |
| 队列任务数 | 异动提醒 | `/alert-rules/events/all` 未确认数 |
| 已调度任务数 | AI洞察 | `/dashboard/stats` → `today_ai_count` |

#### 今日机会榜

- 调用 `/feature/product-rankings` 获取用户商品排名
- 按综合排名排序，展示Top商品
- 每个商品卡片：
  - 商品名+图片
  - 销量百分位（"超过同类87%"）
  - 趋势方向+7天增长率
  - 生命周期阶段标签
  - 竞争力指数
- 点击进入商品详情

#### 异动监控

- 调用 `/alert-rules/events/all` 获取未确认告警
- 调用 `/notifications` 获取监控触发通知
- 按严重级别排序：critical > warning > info
- 每条异动：
  - 告警类型图标（价格下跌/销量激增/评分下降）
  - 商品名+触发指标值
  - 发生时间
  - 一键确认/查看详情

#### 品类热力图+市场动态

- 调用 `/feature/crowd/category-heatmap` 展示品类热度
- 调用 `/feature/crowd/behavior-patterns` 展示行为模式
- 调用 `/feature/crowd/trend-timeseries` 展示趋势曲线
- 调用 `/dashboard/trend` 展示7天/30天趋势

### 3.3 商品页重构：决策面板

> **注意**：ProductDetailView已实现排名百分位和匿名基准的基础调用（fetchRanking/fetchBenchmark），
> 重构重点是增强展示效果和增加快捷操作，而非从零开发。

- 调用 `/feature/product-ranking/{id}` 获取排名百分位 ✅已实现
- 调用 `/feature/anonymous/price-benchmark` 获取价格基准 ✅已实现
- 调用 `/feature/anonymous/sales-benchmark` 获取销量基准 ✅已实现
- 商品卡片显示（需增强）：
  - "你的价格在同类商品中排名前XX%"
  - "你的销量超过同类XX%的商品"
  - 生命周期阶段标签
  - 趋势方向+增长率
  - 快捷AI分析按钮

### 3.4 AI页重构：决策助手

- 保留现有分析记录+报告功能
- AI推荐区域（**Phase 2延后**，需服务端 `/ai/recommendations` API支撑）：
  - 基于排名变化推荐"这3个商品建议做趋势评分"
  - 基于告警事件推荐"这个商品价格异动，建议做风险分析"
  - 基于品类热力图推荐"XX品类正在爆发，建议做选品分析"
- Phase 1 仅做"快捷分析入口"：根据用户已有商品提供分析类型快捷按钮
- 分析结果展示优化：
  - 趋势评分用可视化仪表盘
  - 爆品预测用潜力等级徽章
  - 风险预警用红黄绿三色卡

### 3.5 告警系统统一

- 客户端监控规则页改为调用 `/alert-rules` API
- 告警事件统一从 `/alert-rules/events/all` 获取
- 本地 `LocalEvaluator` 仅作为离线降级方案
- 通知中心整合：监控触发+告警事件+AI完成
- **首页异动监控区**仅展示未确认告警（只读），规则CRUD保留在告警中心页

### 3.6 冷启动与空状态设计（关键补充）

服务端排名/基准/热力图数据依赖 `CloudFeatureEngine.compute_all_rankings()` 计算，
新用户或数据量少的用户可能面临空数据问题。

| 场景 | 空状态处理 |
|-----|----------|
| 排名数据为空 | 显示"数据积累中，添加更多商品后自动生成排名"，引导添加商品 |
| 品类热力图无数据 | 显示"需连接云端服务并积累数据"，引导配置同步 |
| 匿名基准数据不足 | API返回 `{"benchmark": null, "reason": "数据不足"}`，前端显示"同类商品数据不足，暂无法生成基准" |
| 告警事件为空 | 显示"暂无异动，一切正常" + 绿色安全图标 |
| 首页整体无数据 | 显示引导流程："1.添加商品 → 2.采集数据 → 3.查看洞察" |

### 3.7 离线降级策略（关键补充）

采用"本地优先 + 云端增强"策略，确保离线时首页不会空白：

| 数据 | 本地来源 | 云端来源 | 离线行为 |
|-----|---------|---------|---------|
| 机会商品数 | productStore.productCount | /feature/product-rankings | 显示本地商品数，标注"连接云端查看排名" |
| 今日趋势 | 本地最近两次采集对比 | /dashboard/stats → today_trend | 显示本地计算趋势或"-" |
| 异动提醒 | 本地风控告警 | /alert-rules/events/all | 显示本地风控告警 |
| AI洞察 | 本地AI分析记录数 | /dashboard/stats → today_ai_count | 显示本地分析数 |
| 机会榜 | 无 | /feature/product-rankings | 显示"需连接云端" |
| 品类热力图 | 无 | /feature/crowd/category-heatmap | 显示"需连接云端" |

### 3.8 首页聚合API（新增建议）

首页加载需调用8个API，网络往返过多。建议服务端新增聚合接口：

```
GET /dashboard/home
```

一次返回所有首页数据：
```json
{
  "stats": { "opportunity_count": 5, "today_trend": "+12.3%", "alert_count": 3, "today_ai_count": 2 },
  "opportunity_rankings": [...],
  "alert_events": [...],
  "category_heatmap": [...],
  "behavior_patterns": {...},
  "trend_timeseries": {...},
  "dashboard_trend": {...},
  "recent_activities": [...]
}
```

Phase 1 先用并行API调用，Phase 2 实现聚合接口优化。

| `GET /alert-rules/stats/summary` | 告警统计 | 页面加载 |

## 六、与V1.0方案的关键差异

| 维度 | V1.0方案 | V2.0方案 |
|-----|---------|---------|
| 数据来源 | 假设没有全平台数据 | 服务端有CloudFeatureEngine |
| 风险系统 | 从零设计 | 服务端已有AlertRuleEngine |
| AI能力 | 低估 | 8种分析+4种报告+规则引擎兜底 |
| 客户端定位 | 想做全平台扫描 | 决策终端，采集+计算在服务端 |
| 首页设计 | 凭空设计 | 基于服务端已有API |
| 实施难度 | 需要大量新API | 利用已有API，改动集中在前端 |
| FOMO | 担心无数据支撑 | 真实排名/基准/热力图数据驱动 |
